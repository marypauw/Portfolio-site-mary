Homepage.com

Homepage.com is a responsive web application that provides a platform for people interested in buying a property. The application allows users to look for a property, create alerts for a specific kind of property and use a cost calculator to compare prices and make the best choice.

After an initial stint of UX research, the intention of this project was to focus more on the UI development of the product and final prototype with interactions and animations. Prior to this it has been submitted to user testing and A/B testing for further improvement.

01

Understanding the problem

Real estate investment is an increasingly popular way for individuals to achieve financial security. It is an exciting and emotional experience but often complicated. While there are plenty of blogs and agencies providing information, often buyers new to the market may struggle to get started without professional guidance and waste time viewing properties out of their range.

The problems that I decided to face first were:

  • Create a platform where users can have an easy and clear overview of the property on the market
  • Users should be able to apply filters to their research to find something that matches their needs
  • Create a user profile section where users can access saved properties and activate alters for research they are interested in.

 
02

User Persona

To picture my potential audience I decided to create the main user persona, Rashida, which helped me to address the product to a specific kind of user and to create scenarios for future usability testing on the prototype. The web app is meant to be primarily addressed to new, small-scale property buyers who are looking to invest for additional income or financial security. Buyers will use this tool at home or on the go. Users can search for properties anywhere so long as long as they`re logged in on a device. The final goal is that potential users like Rashida get a feel for a place by viewing comprehensive information about the property and its neighborhood before spending time on-site.

 
03

User flows

To structure the initial architecture of the product, I created some paper and digital user flows to understand how to design the first low-fidelity prototype of my website.

I created two different user flows:

  • One specific for the research of a property and to have access to the property information
  • A second one is more focused on the profile section where the users can have access to saved properties and alerts

Wireframes

Low fidelity wireframes

I approached the design process by creating some low-fidelity wireframes with pencil and paper and developed the first ideas about how to solve the problem and how the users could navigate through the website. At this point, I was able to create the first low-fidelity prototype with the basic functionalities of the website. I decided to develop the prototype by redesigning my low fidelity wireframes with Balsamiq and proceed with a first round of usability testing (see description in the next chapter).

see lowfidelity prototype
Mid fidelity wireframes

After establishing a grid system of 12 columns I started designing the first mid-fidelity wireframes defining an ordered system for the organization of the design file. Then giving a first UI identity to the main components, and defining visual hierarchy and spacing based on where I would like to address the user, then empathizing elements which are more important to a specific user flow.

Grid system:

  • Mobile 360px
    Columns nr. 12; Gutter width 8px; Column width 20px; margins 16px;
  • Tablet 768px
    Columns nr. 12; Gutter width 15px; Column width 46px; margins 25px;
  • Small laptop 1200x
    Columns nr. 12; Gutter width 15px; Column width 76px; margins 61px;

High fidelity wireframes

The first step towards the high fidelity wireframes was to gather some inspiration by creating a mood board based on different ideas on the web (and non) which would have helped me to establish a first feeling for the brand identity of the product. The next steps were to define the typography, color palette, imagery, and illustrations to further develop the UI of Homepage.com. The final decisions have been supported by A/B testing through UsabilityHub.com.

06

Different breakpoints

For the project Homepage.com I decided to use three different breakpoints:

  • Mobile 360px
  • Tables 768px @media only screen and (min-width: 768px)
  • Desktop 1200px @media only screen and (min-width: 1200px)

The decision is based on the fact that 360px (according to statcounter.com) is the smallest breakpoint for mobile mostly used. The next two breakpoints are based on Bootstrap's recommended media queries. In every breakpoint, I decided to use the 12 column grid system to make the design as responsive as possible.

08

Testing

The prototype has been tested twice in different stages of the design process, one early-stage with a low fidelity prototype, and a second time with a more defined high fidelity prototype. The final goal of the tests was to know about the learnability and error evaluation of the new user using the product for the first time. The final result of the test helped me to improve the design and the navigation of the product and base my design decision on a user-centred design system.

Main problems highlighted during the usability testing:

  • h4 and text input fields are too small- Need to be increased from 12px to 14px
  • The double menu on the left and right is confusing. Users would search for the profile information in the hamburger menu
  • Layout of the property page needs to be improved
  • Some users would prefer to set their alerts from the filter selection instead in their profile

The only point that I decided to not change for the moment is the second because I think that users would be confused only the first time they use the product but then remember the navigation pattern when they will come back. This point will need further testing in the future.

 
09

The solution

The final Homepage.com prototype present the following functionality:

  • Map and lists overview of the properties on the market
  • Search results can be filtered based on the different criteria such as the number of bedrooms and bathrooms, size, prices, etc.. and the user can create its own research preference which will influence the data retrieval from the backend point of view.
  • Users have access to their own profile section where they can research preferences, have an overview of the saved properties, and set up alerts.

The biggest learning that I gathered from this project are:

  • Users give a lot of importance to a map overview when they can potentially have one so I should always consider it when the product allows it.
  • Animation and illustrations create a lot of engagement and affections to a specific product when wisely implemented.

what's next

The Homepage.com project has been developed as a mobile-first design project, therefore the mobile prototype will be the first one to be handed over to the developers.

I believe that the following parts of the product could still be improved:

  • Further development of the prototype and interactions for desktop and tablet breakpoints
  • Implementation of new elements in the style guide which should always be updated
  • Further testing regarding the access to the alerts sections and how to step them up
  • New functionality compare costs between property to make the best choice
